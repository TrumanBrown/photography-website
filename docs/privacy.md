# What's in the repo about you (and what isn't)

> If you forked this or are auditing it, here's exactly what becomes public when you push, what stays on your machine, and what only lives in cloud-managed vaults.

## What IS public after you push

Anything committed to git is public the moment you push. This repo deliberately keeps the public surface small.

| What | Where | Why it's okay |
|---|---|---|
| **Your full name** | [site.config.ts](../site.config.ts) `ownerName` | Rendered into the footer + EXIF copyright on every photo — the entire point of a personal portfolio is to attribute it to you. |
| **Your apex domain** | [site.config.ts](../site.config.ts) `domain` | Public anyway — anyone can `whois` it. |
| **Your GitHub username** | [infra/main.parameters.json](../infra/main.parameters.json) `githubOwner` | Already public on the GitHub profile that owns the repo. |
| **The Azure region** | [infra/main.parameters.json](../infra/main.parameters.json) `location` | Not sensitive. |
| **The storage account hostname** | [site.config.ts](../site.config.ts) `blobHost` (after first deploy) | Public-by-design — the lightbox uses it to fetch full-res images. |

That's the entire list. **No address, no phone, no email, no subscription/tenant IDs, no keys.**

## What is in this repo about *the project* (not you)

Open source code, design decisions, architecture diagrams. Anyone reading the repo can:
- Reproduce the architecture
- See what Azure services we provision
- Read every workflow YAML
- See the security headers
- See cost estimates

None of that is sensitive — it's documentation by design.

## What is gitignored (lives on your machine only)

| What | Where | Why hidden |
|---|---|---|
| **WHOIS contact info** | `~/.config/photography-website/contact.json` (deliberately outside the repo) | Your real name, address, phone, email. Used once during `az appservice domain create`. Even if you accidentally put it in the repo, [.gitignore](../.gitignore) lines `contact.json` and `contact.*.json` would block it. |
| **Photos themselves** | `src/content/sessions/<slug>/images/*` | Downloaded by the prebuild from Blob; never committed. |
| **Per-session sidecar JSON** | `src/content/sessions/*.json` | Same — regenerated every build from `_session.json` in Blob. |
| **Built site** | `dist/` | Regenerated by `npm run build`. |
| **Astro type cache** | `.astro/` | Tooling artifact. |
| **Local prebuild cache** | `.cache/` | Downloaded blob copies, keyed by ETag. |
| **`node_modules/`** | top-level | npm install output. |
| **`.env*`** | anywhere | Reserved. We don't use env files today, but if you ever add one it's pre-blocked. |

## What lives in **GitHub Actions secrets** (vault, never in code)

These are real secrets. They're set in **GitHub → Settings → Secrets and variables → Actions**, never in a file.

| Secret | Sensitive? | Source |
|---|---|---|
| `AZURE_CLIENT_ID` | Identifier only | Output of `scripts/setup-federated-credential.sh` |
| `AZURE_TENANT_ID` | Identifier only | Same |
| `AZURE_SUBSCRIPTION_ID` | Identifier only | Your Azure subscription |
| `AZURE_STATIC_WEB_APPS_API_TOKEN` | **Yes — real secret** | Bicep deploy output (`scripts/bootstrap-swa-token.sh` writes it) |
| `AZURE_STORAGE_ACCOUNT` | Identifier only | Bicep deploy output |
| `APPINSIGHTS_CONNECTION_STRING` | Mildly — includes a write key | Bicep deploy output |
| `GH_PAT_FOR_SECRETS` (optional) | **Yes — real secret** | A fine-grained PAT you create manually |

The three "identifier only" Azure IDs are technically not secrets in Microsoft's threat model — knowing them doesn't grant access. They're stored as secrets out of tidiness.

The only true credentials with broad blast radius are:
- The SWA deploy token (rotated by re-running the Bicep deploy)
- Optionally, your GH PAT (you decide whether to even create it)

## What lives in **Azure** (not in this repo, not in GitHub)

| Thing | Where |
|---|---|
| Your domain registration's WHOIS record | Azure's registrar partner (GoDaddy under the hood). Privacy protection on by default → your real address is hidden from public WHOIS lookups but still on file with the registrar. |
| The bootstrap managed identity's principal ID | Azure RBAC tables. |
| Your actual photos | The `originals` + `derivatives` containers in Blob Storage. **Anyone who knows the URL can read individual blobs.** The container itself cannot be listed by anonymous users — there's no directory index, so URLs cannot be enumerated. |
| Site analytics (pageviews) | Application Insights → Log Analytics workspace. App Insights snippet respects `navigator.doNotTrack`. |

## What is published to **the public website**

| What | Where |
|---|---|
| Your name (footer + meta tags + EXIF) | Every page |
| Every photo you upload | The relevant gallery page + as a lightbox URL pointing at Blob |
| The site's full set of pages | Indexable by search engines (`robots.txt` allows it) |

## Specifically NOT collected, exposed, or stored

- No visitor IP addresses are stored (Azure CDN logs them only briefly for security; not surfaced to you).
- No visitor cookies.
- No visitor accounts (there are no accounts).
- No third-party analytics scripts (just Microsoft's own App Insights).
- No third-party ad networks.
- No payment info.

## TL;DR

**Public:** your name, your domain, your GitHub username, your storage hostname.
**Private (lives only on your machine):** your address, phone, email.
**Private (lives only in GitHub vault):** the SWA deploy token.
**Private (lives only in Azure):** your photos' file URLs (technically guessable if someone knows naming patterns; not enumerable).

If you ever change your mind about your name being visible, edit [site.config.ts](../site.config.ts) — it's a one-line change → commit → next build redeploys with the new value.
