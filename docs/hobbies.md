# Hobbies section

A second, **optional** top-level section beside Photography. Photography is unchanged and remains the home page (`/`) and default landing. Hobbies lives at `/hobbies` with one page per hobby, each able to mount a self-contained interactive "island."

This doc covers how the section is wired, how to add a hobby, the conventions every interactive follows, and the aquarium island as the reference implementation. It is written so a future session (or you) can pick the work back up cold.

> Status: shell + the **Aquarium Keeping** pilot are live. Four more islands are planned (travel map, WA fishing map, pixel hike, repo explorer) and meant to be built **one at a time**.

| Nav + sidebar behavior | [src/components/Header.astro](../src/components/Header.astro), [src/layouts/BaseLayout.astro](../src/layouts/BaseLayout.astro) |

Key differences from the Photography (`sessions`) collection:

- Hobbies JSON is **hand-authored and committed** to git. Sessions are gitignored and generated by `scripts/prebuild.mjs` from Blob Storage. There is **no prebuild, no Blob, no Azure** involvement for hobbies.
- A hobby page does **not** show the sessions sidebar. `BaseLayout` takes a `section` prop (`'photography'` default | `'hobbies'`); when `'hobbies'` it hides `SessionNav` and its mobile hamburger, and the Header highlights the Hobbies nav item.

The `sections.hobbies` flag only controls the **header nav link** (discoverability). The pages still build when it's off — turning it off "ships dark," it doesn't remove the routes.

## Adding a hobby

1. **Create the content entry** `src/content/hobbies/<slug>.json`:

   ```json
   {
     "title": "Hiking",
     "summary": "Short blurb shown on the hobby card.",
     "intro": "A sentence or two shown on the hobby page above the interactive.",
     "emoji": "🥾",
     "accent": "#16a34a",
     "order": 3,
     "interactive": "pixel-hike"
   }
   ```

   - `slug` comes from the filename (`hiking.json` → `/hobbies/hiking`).
   - `emoji` + optional `accent` (hex) render the card icon tile — keeps the grid asset-free.
   - `order` sorts the landing grid (ascending); entries without `order` fall to the end, alphabetical.
   - `interactive` is one of the enum values in the schema, or omit it for a static page.

2. **(Optional) build the interactive island** — see the pattern below. If you only want a static page (text/photos), skip this and leave `interactive` unset.

3. **Wire the island** in [src/pages/hobbies/[slug].astro](../src/pages/hobbies/%5Bslug%5D.astro): import the component and add a branch:

   ```astro
   {interactive === 'pixel-hike' && <PixelHike />}
   ```

That's it. No build-config, no Blob, no CI changes.

## Interactive island conventions

Every island follows the same rules so they stay cheap, safe, and consistent with the rest of the site:

1. **Self-contained & offline.** No runtime APIs, no map tiles, no third-party scripts, no remote data. All data is baked in; all graphics are drawn procedurally (canvas/SVG) or are tiny local assets. This keeps them free, fast, and **zero-maintenance** — once built you never feed them data (unlike photos).
2. **CSP-safe by construction.** Use Astro's processed `<script>` (with `import`), **never** `is:inline`. Astro emits it as a same-origin module under `/_astro/…js`, which satisfies the strict `script-src 'self'` policy in [staticwebapp.config.json](../staticwebapp.config.json). No per-script hashes to maintain. (Confirmed in the build output: the aquarium ships as `/_astro/AquariumTank.astro_…js`.)
3. **Lazy by route.** An island's script only loads on its own `/hobbies/<slug>` page, so Photography pages stay zero-JS and the build stays light.
4. **Mobile-first-ish.** Works on touch, even if richer on desktop — same bar as the photography section. Responsive sizing (`ResizeObserver`), touch input (`pointerdown`, native range inputs), and pausing offscreen.
5. **Minimal dependencies.** Prefer vanilla + canvas/SVG. Add a small library only if it *meaningfully* improves graphics/accuracy — and document it here.

Markup lives in a `.astro` component under `src/components/hobbies/`; non-trivial logic lives in `src/lib/hobbies/<name>.ts` and is imported by the component's `<script>`. The component binds to the engine via `data-*` hooks, so the engine owns behavior and the markup stays static.

## Reference implementation: the aquarium island

A planted-tank **stocking sandbox**. Tap a species to add it, resize the tank, and watch a bioload meter react. It is meant to be fun and roughly believable, not a real stocking calculator.

**Files**

- [src/components/hobbies/AquariumTank.astro](../src/components/hobbies/AquariumTank.astro) — the markup (canvas + controls), the mounting `<script>`, and the "My tank" photo section.
- [src/lib/hobbies/aquarium.ts](../src/lib/hobbies/aquarium.ts) — the whole engine: species data, the stocking math, the canvas pixel rendering, and the UI sync. **Zero dependencies.**

**Single source of truth.** The `SPECIES` array in `aquarium.ts` (~35 freshwater species) drives everything. The palette buttons and the stocking list are built from it at runtime, so the markup carries no species data. Each entry stores an approximate adult length (inches), a relative `bioload` weight, a sensible `minGallons`, a swim `zone` (top/mid/bottom), a body `shape`, and two colors.

### How the stocking math works

Two tunable constants at the top of `aquarium.ts` plus one sum:

```
capacity = gallons × PLANTED_CAPACITY_PER_GALLON     // 0.55 "load units" per gallon
load%    = Σ(bioload of every fish in the tank) / capacity
```

So a 20-gallon tank has a capacity of `20 × 0.55 = 11` units. Seven neon tetras at `0.3` each (`2.1`) plus four pygmy cory at `0.35` (`1.4`) plus five cherry shrimp at `0.05` (`0.25`) plus one honey gourami at `0.9` come to `~4.65`, or about 42% of capacity, which reads **Healthy**.

The meter colors and labels come straight from `load%`:

| Load | Label | Color |
|---|---|---|
| `< 40%` | Lightly stocked | green |
| `40–85%` | Healthy | green |
| `85–100%` | Heavily stocked | amber |
| `> 100%` | Overstocked | red |

Separately, each species carries a `minGallons`. If the current tank is smaller than any stocked species likes, a note appears (for example "Angelfish likes 29+ gallons"). That check is independent of the bioload percentage, because some fish need swimming room regardless of waste.

### Where the numbers and species come from

Being upfront, because the request was that anyone can learn how this works:

- **The species roster is hand-picked from general fishkeeping knowledge** (the fish people actually keep in planted community tanks), not pulled from any database, API, or dataset.
- **The per-species numbers are approximate and tuned by feel.** The adult sizes are ballpark hobby figures; the `bioload` weights roughly track how big and messy a fish is; the `minGallons` values are common community recommendations. None of them are precise or authoritative, and the bioload weights are not derived from measured waste data — they were chosen and play-tested until the meter behaved sensibly.
- **The model is a miniature of real stocking heuristics.** The old rule of thumb is "one inch of fish per gallon," and community tools such as AqAdvisor estimate stocking from a large species database. This sandbox uses the same idea with a single "load per gallon" constant instead of a database. The `0.55` figure and the 40 / 85 / 100% thresholds were picked so a normal community tank reads healthy and obvious overstocking turns red.
- **Bottom line: it is a toy.** If you are stocking a real tank, research each species properly rather than trusting this meter.

### How the pixel art is made

Everything you see in the tank — fish, plants, rocks, driftwood, bubbles, light — is **drawn in code** with canvas `fillRect` calls. There are no image files, sprite sheets, stock photos, or AI-generated art anywhere in the scene, so there is nothing to license or attribute. (The only image is the "My tank" placeholder SVG, which you replace with your own photo.)

- **The pixel look** comes from a fixed block size `PIX` (4 CSS pixels). Every shape snaps to that grid and image smoothing is turned off, which gives the chunky 8-bit feel. Dropping `PIX` makes finer pixels; raising it makes chunkier ones.
- **Fish are procedural.** Each one is built from its species params: a body drawn as horizontal pixel rows of an ellipse (`fishBody`), a triangular `fishTail`, a lateral stripe or vertical bars, dorsal and pectoral fins, and an eye. A small `shade()` helper mixes the body color toward white for the top rows and toward black for the bottom rows, which fakes a bit of lighting. There are five silhouettes — `fish`, `tall` (gouramis/cichlids), `long` (loaches/plecos), `shrimp`, and `snail` — and fish are drawn back-to-front by depth `zone` for layering.
- **The scene is layered for performance.** A **baked** offscreen background (water gradient, dark silhouette plants for depth, four rocks, two driftwood pieces, gravel substrate, and a carpet of plants along the front) is rendered once and only redrawn on resize. Each animation frame just blits that image and then draws the moving parts: slanted caustic light, two swaying foreground plant types (tall vallisneria and stems), fish, floating surface plants with dangling roots, and rising bubbles.
- **Layout is seeded, not random each frame.** Plant, rock, and bubble positions come from a tiny seeded pseudo-random generator (`makeRng`) keyed to the tank width, so the scene is stable but adapts when the canvas resizes.
- Meter bar colors are set as **inline styles**, not Tailwind classes, because Tailwind's build step would purge class names it can't see in the source.

### Mobile and performance

- The canvas resizes to its container via `ResizeObserver`; height clamps to 220–460px and fish scale with canvas width.
- Because the static background is baked and blitted, only fish, foreground plants, floaters, and bubbles are recomputed per frame.
- The loop pauses when the canvas scrolls offscreen (`IntersectionObserver`) or the tab is hidden (`visibilitychange`), so it costs nothing when not in view.
- Touch friendly: native range slider, tappable palette buttons, and tap-to-add (after picking a species, tapping in the tank drops one at that point).
- `prefers-reduced-motion` slows everything and stops the plant sway.
- A hard cap of 55 fish keeps the loop smooth on phones.
- It all runs in the visitor's browser, so frame rate depends on their device, not on the hosting tier.

### Tweaking it later

- Species roster and balance: the `SPECIES` array.
- Overall stocking feel: `PLANTED_CAPACITY_PER_GALLON`.
- Starting community shown on load: `STARTER`.
- The look of anything: the `draw*` / `bake*` functions and the `PIX` constant.
